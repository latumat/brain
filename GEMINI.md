# Brain — LLM Wiki Schema

You are my LLM Wiki agent. This vault is a persistent, compounding second brain. You maintain it; I curate and direct.

## Directory Layout

```
/
├── GEMINI.md           ← this file — your operating instructions
├── raw/                ← immutable source documents (you read, never modify)
│   └── assets/         ← locally downloaded images referenced by sources
└── wiki/               ← everything you write and maintain
    ├── index.md        ← content catalog (update on every operation)
    ├── log.md          ← append-only operation log
    ├── overview.md     ← evolving high-level synthesis (create when enough sources exist)
    ├── sources/        ← one summary page per ingested source
    ├── entities/       ← people, organizations, products, places
    ├── concepts/       ← ideas, frameworks, theories, processes
    ├── topics/         ← broader subject areas that aggregate entities + concepts
    └── outputs/        ← analyses, comparisons, slide decks, charts generated by queries
```

## Page Format

Every wiki page starts with YAML frontmatter:

```yaml
---
title: "Page Title"
type: source | entity | concept | topic | output | overview
tags: [tag1, tag2]
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources: 0          # number of raw sources this page draws from
---
```

Then use standard markdown. Use `[[wikilinks]]` for all cross-references — never bare file paths.

### Source Summary Pages (`wiki/sources/`)

Structure:
1. **One-line summary** — what this source is and why it matters
2. **Key takeaways** — bulleted list of the most important points
3. **Entities mentioned** — links to entity pages
4. **Concepts mentioned** — links to concept pages
5. **Contradicts / Supports** — notes on how this source relates to existing wiki claims
6. **Raw source** — link to the file in `raw/`

### Entity Pages (`wiki/entities/`)

Structure:
1. **What it is** — one paragraph definition
2. **Key facts** — bulleted list
3. **Related entities** — wikilinks
4. **Related concepts** — wikilinks
5. **Sources** — wikilinks to source pages that mention this entity

### Concept Pages (`wiki/concepts/`)

Structure:
1. **Definition** — clear, concise
2. **Why it matters** — practical significance
3. **Key properties / variants** — if applicable
4. **Related concepts** — wikilinks
5. **Examples** — real instances, entities that exemplify this concept
6. **Sources** — wikilinks

### Topic Pages (`wiki/topics/`)

Broader aggregations. Structure:
1. **Overview** — 2–3 paragraph synthesis
2. **Key entities** — wikilinks
3. **Key concepts** — wikilinks
4. **Open questions** — things not yet resolved in the wiki
5. **Sources** — wikilinks

### Output Pages (`wiki/outputs/`)

Analyses, comparisons, tables, charts generated in response to queries. Always file these back — good answers shouldn't disappear into chat history.

## Workflows

### Ingest

When I give you a source to ingest:

1. Read the source file from `raw/`
2. Discuss key takeaways with me briefly (2–4 bullet points, ask if I want to redirect emphasis)
3. Write a summary page in `wiki/sources/` — filename: slugified title (e.g. `the-lean-startup.md`)
4. For each significant entity mentioned:
   - If the entity page exists: update it, add this source to its Sources section
   - If not: create the entity page
5. For each significant concept mentioned:
   - If the concept page exists: update it
   - If not: create the concept page
6. Update any relevant topic pages
7. Note contradictions with existing wiki claims explicitly on both pages
8. Update `wiki/index.md` — add the new source and any new pages
9. Append an entry to `wiki/log.md`

A single source will typically touch 5–15 wiki pages. That's normal and good.

### Query

When I ask a question:

1. Read `wiki/index.md` to find relevant pages
2. Read the relevant pages
3. Synthesize an answer with citations (`[[page]]` wikilinks)
4. If the answer is non-trivial (comparison, analysis, discovery), ask if I want to file it as an output page in `wiki/outputs/`
5. Append a query entry to `wiki/log.md`

### Lint

When I ask you to lint the wiki:

1. Read all pages (use the index)
2. Report:
   - Contradictions between pages
   - Stale claims (flagged with newer source)
   - Orphan pages (no inbound links)
   - Concepts/entities mentioned but lacking their own page
   - Missing cross-references
   - Data gaps that could be filled with a web search
3. Suggest new sources to look for
4. Suggest new questions to investigate
5. Append a lint entry to `wiki/log.md`

## Index Conventions

`wiki/index.md` is content-oriented. It is the LLM's navigation tool. Structure:

```
## Sources (N)
- [[sources/title]] — one-line summary

## Entities (N)
- [[entities/name]] — one-line summary

## Concepts (N)
- [[concepts/name]] — one-line summary

## Topics (N)
- [[topics/name]] — one-line summary

## Outputs (N)
- [[outputs/name]] — one-line summary
```

Always update counts. Always keep entries alphabetically sorted within each section.

## Log Conventions

`wiki/log.md` is append-only. Each entry format:

```
## [YYYY-MM-DD] <operation> | <title>

<1–3 line summary of what happened>
Pages touched: [[page1]], [[page2]], ...
```

Operations: `ingest`, `query`, `lint`, `create`, `update`

Never delete or edit past log entries.

## General Rules

- You write all wiki files. I do not write them (unless I explicitly say so).
- Never modify files in `raw/`. They are the immutable source of truth.
- Always update `index.md` and `log.md` at the end of every operation.
- Use wikilinks (`[[...]]`) for all cross-references, never bare paths.
- When a new source contradicts an existing wiki claim, note the contradiction on both the source page and the affected wiki page — don't silently overwrite.
- Prefer updating existing pages over creating new redundant ones.
- Keep summaries tight. Bullet points over prose where possible.
- If a page grows unwieldy (>200 lines), suggest splitting it.
- Ask me before creating more than 10 new pages in one operation — large rewrites should be intentional.
